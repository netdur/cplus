CONVERGED MODEL — ownership, mutability & the mental anchor
[[SUPERSEDED: the 3-tier const/let/var below was dropped 2026-06-18.
 Final model = 2x2 const(module value)/static(mutable C global)/
 let(immutable local)/var(mutable local), no `mut`; the middle
 "frozen-binding, mutable-fields" let tier is gone. `ref` needs a
 `var` place. See plans/plan.md.]]
(supersedes the `mut -> var/ref` split entry below: bindings are
 const/let/var, params/receivers use bare/ref/take, `borrow` is DELETED)
================================================================

MENTAL ANCHOR: a C+ struct is a C++ CLASS WITH CONST-CORRECTNESS —
but no inheritance, no virtuals (use composition + `interface`).
`const a; a.bump()` is illegal for exactly the C++ reason: you can't
call a non-const (mutating) method on a const object. This anchor is
heavily represented in LLM training data and yields correct C+ for
free — a far better anchor than Rust's `&self`/`&mut self`.

TWO axes. (1) BINDING mutability — your own variable, three tiers:

  const a   frozen          no rebind, no field write
  let  a    fields mutable  no rebind; `a.field = ..` ok     [= TypeScript const]
  var  a    fully mutable   rebind within the declared type + fields; scalar reassign

(2) VALUE relation — read / mutate / consume — for receivers and params:

  surface     read     mutate      consume
  receiver    this     ref this    take this
  param       x: T     ref x: T    take x: T

`var` (mutable binding) and `ref` (mutate) hit DIFFERENT targets: `var`
makes YOUR binding writable; `ref` writes back to the CALLER's value. A
`ref` arg's caller place must allow the write the callee performs: `const`
never; `let` only for interior/content mutation like `c.n = 3`; `var` is
required when the callee assigns the parameter place itself like `c = 3`
(including scalar write-back).

BINDINGS (const / let / var — THREE tiers, the user's original model):
- `const a` = frozen binding and frozen contents: no rebind, no field write (`a.B = 3` illegal). C's `const`.
- `let a` = immutable binding, mutable contents (= TypeScript `const` / normal OOP object reference): no reassign / reshadow (`a = {..}` illegal), but fields ARE mutable (`a.B = 3` legal).
- `var a` = mutable binding and mutable contents: rebind to another value of the SAME declared type AND mutate fields (`a = C {..}` and `a.B = 3` both legal; `a = Other {..}` illegal); a scalar `var i` reassigns (`i = i + 1`).
- For a SCALAR, `let` and `const` coincide (no fields -> frozen); the mutable scalar is `var`. (History: I briefly argued to collapse to const/let and called `let` "fully mutable", dropping `var` — a terminology slip; the user's `let` is the MIDDLE tier, so the fully-mutable `var` is still needed. Reverted to const/let/var 2026-06-18.)
- `ref` caller-place rule: a `ref` arg needs a place that permits the actual write — `const` never; `let` only if the callee mutates interior contents/fields (`c.n = 3`); `var` if the callee assigns the parameter place itself (`c = 3`), including scalar write-back.
- Impl note: today `const` is module-only / compile-time / literal-init and does NOT parse as a local (verified) — generalize to a runtime local; `var` is the existing mutable-binding lowering (was `let mut`).

RECEIVERS (this / ref this / take this):
- `this` = read (const method). `ref this` = mutate (non-const method).
  `take this` = consume. `ref` is the MODIFIER; the name is always `this`,
  so field access is `this.B`, never `ref.B`.
- A `ref this` (mutating) method on a `const` binding is rejected (E0328,
  already enforced); on a `let` binding it is legal.

PARAMS (bare / ref / take — `borrow` deleted, see below):
- bare `x: T` = read-only borrow, every type.
- `ref x: T` = by-reference write-back (any type; caller place must permit
  the write: `const` never, `let` for interior content/field mutation,
  `var` for assigning the parameter place itself / scalar write-back).
  No call-site `&`.
- `take x: T` = take ownership; an owned value is MUTABLE by default
  (`take v: Vec; v.push(..)` just works — the old move+mut mutual-exclusion
  gap is closed by mutable-by-default).

VISIBILITY TRADEOFF (accepted): with no call-site `&`, mutation is not
visible at the call. `const` is the opt-in "this won't change" signal
(both C and Rust instead mark it at the call: `&C` / `&mut x`). This is
the C / TypeScript philosophy — convenient by default, opt into safety.

RECEIVERS & CONSTRUCTION (decided): EXPLICIT receivers — OOP injected-`this`
is REJECTED (despite being the bigger LLM-familiarity win). Injecting `this`
would force C++-style const/mut method modifiers, a `static fn` marker for
receiver-less functions, break the unified bare/ref/take table, and leaves no
clean spelling for `take this`. So receivers stay in the signature —
`this` / `ref this` / `take this` — and a receiver-less `fn make()` is just an
associated function. CONSTRUCTION uses ONE primitive: the struct literal
`A { B: 0 }` (C brace-init, now usable cross-module thanks to public fields).
Add type-INFERRED literals so the type isn't repeated: `let a: A = { B: 0 };`
and `return { B: 0 };` (verified: this does NOT parse today — small parser/sema
add). `new` is a PLAIN factory function, nothing special:
`fn new() -> A { return { B: 0 }; }`. No OOP constructor, no injected `this`,
no special `init` — that would be a second construction mechanism, against the
"no several ways" principle.


================================================================
RENAME — inverted or wrong semantics, these actively mislead LLMs
================================================================

# impl
extend  [[SUPERSEDED 2026-06-18: `extend` REVERTED — keep `impl`, change only the connector `for`->`:` => `impl T: I` / `trust impl T: Send`. `extend` dragged in a Java-`extends` inheritance false friend and overloaded `as` (cast/import-alias). The `:` connector still kills the inverted muscle memory (a `impl Display for Point` Rust prior now hits a syntax error). See plan.md.]]
C+ impl is type-first (`impl Type {}` / `impl Type for Iface {}`) — the exact reverse of Rust's `impl Trait for Type` — so `extend Type {}` / `extend Type as Iface {}` reads type-first and kills the inverted muscle memory.

# self
this
C+ `self` lowers to a pointer (a borrow), exactly like C++ `this`, so `this` carries the correct "I am a pointer, not a moved value" intuition that Rust's by-value `self` destroys; receiver triad: read = `this`, mutate-caller = `ref this`, consume = `take this` (a receiver is never a scratch copy, so `var this` never occurs).

# Self
This
Pairs with `this`; the enclosing-type alias should not wear Rust's `Self` and drag in by-value-self assumptions.

# mut
var / ref
Decided (CONVERGED MODEL): binding mutability is `const`/`let`/`var`; param/receiver mutation is `ref`. `var` is retained as the fully mutable binding tier, replacing today's `let mut`. Why `mut` had to go: `mut x: T` was a false friend — a Copy type meant a by-VALUE copy (caller unaffected, sees 5), a non-Copy type meant by-REFERENCE mutation (caller sees 6), with NO syntactic hint which (Copy-ness is implicit). That ambiguity forced the split; `var` (binding) + `ref` (caller write-back) resolve it.

# ref  (the by-reference mutation keyword)
ref
By-reference write-back, replaces `mut` on params + mutable receivers: `fn bump(ref x: S) { x.v = x.v +% 1; }` with `let x: S = ...; bump(x);` leaves x.v == 6. Matches C#'s `ref`. Works on ANY type (C-ABI: `ref x: S` lowers to `S*`, `ref i: i32` to `i32*` — C's out-param). No call-site marker by design (`bump(x)`, never `bump(&x)`). HARD RULE: `ref` requires a caller place that permits the actual mutation — `const` never; `let` for interior field/content mutation (`x.v = ...`); `var` if the callee assigns the parameter place itself (`x = ...`), including scalar write-back. LIVE HOLE to close: today `mut s: S` silently mutates an immutable source (returns 6, no error — E0305 fires for direct/field writes but not across the param); enforce the rule on `ref` args (writability check already exists; wire it on).

# move
take
As a param/receiver prefix C+ `move` declares callee-side ownership transfer, but Rust `move` means closure capture — `take x: T` / `take this` reads as "consume this" with no closure association; after the flip below, `take` is the SOLE way to give ownership away.

# (bare param)  x: T   — FLIP the default
read-only borrow (every type)
Today a bare `x: T` is inconsistent: a copy for Copy types but a MOVE for non-Copy (Rust-like, and it silently eats the argument). Flip it so bare = read-only, non-consuming borrow for EVERY type — then "bare = readonly" is true uniformly. Verified the safety net already exists: returning a bare Copy yields a copy (works today), and returning OR field-storing a bare non-Copy is blocked by E0337 ("would create a second owner; take by value or clone") — use `take` or an explicit `.clone()`. Implicit non-Copy copy-on-return is deliberately NOT done (hidden malloc vs no-alloc/C-ABI rules; undefined for File/Mutex/Arc). Cost is migration (bare non-Copy flips move->borrow across stdlib/tests), not new compiler analysis.

# borrow  (TWO distinct constructs — DELETE BOTH)
DELETE
(1) PARAM PREFIX `borrow x: T` — shared, non-consuming borrow. Subsumed by the bare read-only default above; the 58 real call sites (vendor/json, vendor/metal, vendor/coreai, vendor/agent_core, docs/examples, ...) just drop the keyword: `borrow x: T` -> `x: T`. `ref` covers mutable-by-ref, `take` covers ownership transfer.
(2) REGION / LIFETIME TYPE `borrow A T` (shared, = Rust `&'a T`) and `mut x: borrow A T` (exclusive, = `&'a mut T`) — named-lifetime references with full region checking (E0511 undeclared region, E0512 region mismatch; the textbook `longest`/`pick` functions). This is the MOST Rust-like construct in the whole language and is used ZERO times in production (the only `.cplus` hit is a bug repro under bugs/). DROP IT ENTIRELY: remove the `Borrowed` TypeKind (threads ast/parser/sema/lower/resolver/monomorphize/codegen/borrowck/graph — 9 files), the region grammar in parser, E0511/E0512 in sema, the 19 e2e region tests, and the doc sections (ERRORS.md, MEMORY-MODEL.md, SPEC.md, errors.toml, design/phase6-borrow-exclusive.md, design/phase7-generics.md, tutorial.md). The C-bounded way to return data is: return owned (`Text`, `Vec`, structs), `take`, or return a blessed view (`str`, slice) only when sema/borrowck can prove the backing storage does not die at the return boundary; otherwise use raw `*T` for explicitly unmanaged views.

# match
match (KEEP — decided)
Keep `match`. v0.0.24 already changes a lot; renaming match too (e.g. to `switch`) adds churn AND brings its own false friends — C `switch` has fallthrough + `case X:` syntax + statement form, none of which C+ has (exhaustive, no-fallthrough, `=>` arms, expression). Net: `switch` would mislead as much as `match`. Document the limits instead (one level deep, wildcard-or-binding payloads, no guards/ranges/or-patterns/bool-literals). Open minor left as a possible later tweak: the mandatory full variant path `Maybe[i32]::Some(v)` could be inferable to `Some(v)` from the matched type (verbosity), but that is a feature, not a rename.

# unsafe
DROP the keyword — decided (reverses the earlier KEEP)
DROP `unsafe` entirely. The enabling insight: C+ has NO references, so the UB-capable operations are already self-flagging in the syntax and can never happen by accident:
- deref: unary `*p` — `*` can ONLY mean raw-pointer deref (nothing else to confuse it with, unlike Rust's `*` over `&`/Box). You cannot deref without typing `*`.
- pointer types / int->ptr: `*T` and `x as *T` are visible in the expression.
- ptr->int: replace the bare `p as usize` cast with a LOUD intrinsic `#addr(p)` [[SUPERSEDED 2026-06-18: `#addr(p)` intrinsic, not a bare `addr()` builtin — matches the `#name()` convention and avoids squatting the `addr` identifier; coexists with `#addr_of(x) -> *T`]], so the pointer op shows in the expression instead of hiding in the variable's type.
This satisfies the identity ("not ACCIDENTALLY") WITHOUT a block keyword — a raw op is never invisible. Ownership accounting (`opaque`/`drop`, E0510) and field privacy (`_` prefix) are INDEPENDENT of `unsafe` and survive unchanged, so safe outside code still cannot corrupt a type's private `_ptr`/`_len`.
SAFETY CLAIM DOWNGRADE (be honest in docs): C+ DROPS Rust's "safe code provably cannot cause UB" (there is no safe/unsafe partition anymore). New claim: "every UB-capable operation is syntactically visible; UB is never accidental" — Zig-class, PLUS C+'s ownership checker, which Zig lacks (so C+ still catches use-after-free at compile time). Net identity: Zig ergonomics + compile-time ownership.
CONSEQUENCES: the `unsafe struct` idea is MOOT (no safe/unsafe boundary to protect) — encapsulation is carried entirely by `_`-private fields (enforced cross-module). `unsafe { }` blocks and `unsafe fn` are removed; the E0801 deref/cast gate is removed (ops now allowed, flagged by `*`/`addr`/`as *T`). `unsafe impl T for Send/Sync {}` becomes `trust impl T for Send/Sync {}`: `trust` is a manual type-author assertion the compiler cannot derive. This is NOT `opaque`: `opaque` only tells the ownership/drop audit that a raw pointer field is managed elsewhere / not freed by this type; it does not imply thread-transfer or sharing safety.
FOREIGN RESIDUE — RESOLVED (option B, declaration-is-the-marker): extern calls and static-mut need NO call-site marker. You cannot use a foreign fn without first DECLARING it (`extern fn free(...)`) or IMPORTING it (which forces a `::` namespace, `c::free`). That declaration/import IS the visible acknowledgment — a foreign call can never appear without one preceding it, so it is never accidental. Calls stay bare (`free(p)` / `c::free(p)`); `static mut`'s declaration is likewise its own marker, access stays bare. The `extern` and `static mut` keywords STAY — they are now the declaration-site flags that replace `unsafe`'s job for the foreign boundary.

# enum
enum (keep)
It is a tagged union (Swift-style), not C's integer enum — keep the mainstream word but document that variant payload patterns are one level deep.

# await
await (keep)
Standard word, but C+ uses prefix `await future` vs Rust's postfix `.await` — keep the word, note the position.

# Copy
Copy (keep) / Trivial
Structural and auto-derived (no `#[derive(Copy)]`) unlike Rust — keep the name with a "no derive needed" note, or rename to `Trivial`/`Bits` to stop LLMs writing derives.

# drop
drop (keep) / deinit
It is a plain method `fn drop(take this)` inside the type's extend block, not an `impl Drop for T` block — keep `drop` with a note, or use `deinit` (Swift) to break the Rust-Drop pattern.

KEEP — Rust-shaped but semantically safe, no false friend

# fn
fn
Function declaration maps cleanly to a C function; the trap is body semantics, not the word.

# let
let (immutable binding, mutable contents — TS `const`; MIDDLE of const/let/var)
`let` is an immutable binding (no reassign `x = ...`, no reshadow) with mutable contents (`x.field = ...` legal) — TypeScript's `const` / normal OOP object reference. For a scalar (no fields) it is effectively frozen; the mutable scalar is `var`.

# const
const (DEEP immutable local binding — top of const/let/var)
`const a: A` = deep immutable — no rebind AND no field write (`a.B = 3` illegal); C's `const`. Contrast `let`, which freezes only the BINDING, not the fields. Today `const` is module-only / compile-time / literal-init and does NOT parse as a local (verified) — generalize to a runtime local.

# var
var (fully mutable binding within the declared type — bottom of const/let/var)
`var a` = rebind to another value of the same declared type AND mutate fields (`a = C {..}` and `a.B = 3` both legal; `a = Other {..}` illegal); a scalar `var i` reassigns (`i = i + 1`). The mutable-binding tier — required for counters/accumulators and as the caller place for a scalar `ref i: i32` write-back. (= today's `let mut` lowering. I briefly advised dropping it; reinstated 2026-06-18.)

# static
static

# pub
DROP as a visibility keyword — `_` prefix replaces it (uniform with fields)  [[OPEN 2026-06-18: settled for FIELDS + METHODS (`_`=private, public by default). For top-level ITEMS (fn/struct/enum) it is UNRESOLVED — one memory note says keep a short `pub`, this entry says drop it; not revisited in the 2026-06-18 session. Decide before implementing.]]
DECIDED (reverses the earlier KEEP): drop `pub` as C+ source visibility. Visibility is UNIFORM — `_name` = private for top-level items, struct fields, AND methods; everything else is PUBLIC to C+ imports by default (the Dart model, compiler-enforced). One rule everywhere (big for LLM-first), drops a Rust-ism, and resolves the items-private-default vs fields-public-default asymmetry. The friction (changing visibility = a rename) is on the RIGHT side: promoting `_foo`->`foo` forces touching every call site = a review checkpoint (you may rediscover why it was private); demoting breaks external callers with compile errors. Migration is a non-issue (pre-1.0, `vendor/` is the whole world). Residual risk: forgetting `_` at birth = silently public (mitigate with a lint — and a publicly-exposed `*T` field is the DEVELOPER'S deliberate choice — C+ does NOT auto-privatize raw pointers, that would be hand-holding; the actual UB op, deref `*`, stays visible regardless, and the exposure was a typed decision). DECIDED: `*T` fields follow the UNIFORM rule (public by default, `_` to hide) — no carve-out. SCOPE: `_` = private only where visibility EXISTS (items / fields / methods); locals have no visibility so `_x` local is just a name; bare `_` stays the wildcard. IMPORTANT SPLIT: the old C ABI/linker/header role of `pub` is NOT inherited by public-by-default C+ items; it moves to `export`.

# export
export
NEW keyword for the C ABI / linker / generated-header surface only. It is NOT a C+ module visibility marker. Plain non-underscore items are already visible to C+ imports, but they are not stable exported symbols and can still use internal linkage / fast calling conventions when codegen proves that is valid. `extern fn malloc(...);` stays a C import declaration. `export extern fn add(...) { ... }` replaces today's `pub extern fn add(...) { ... }`: C ABI checked, externally linked, and emitted in headers. Header generation walks `export` items, not every public-by-default C+ item.

# struct fields: PUBLIC by default (DECIDED) — private via `_` prefix
public-by-default fields; `_name` = private (Dart-style, enforced)
DECIDED (C-ABI is the tiebreaker): struct FIELDS are PUBLIC by default. C+ is C-ABI bound and a C struct exposes all its fields, so the `struct` keyword must mean what it means in C/C++ — a public bag of fields. Privacy is opt-IN via a leading underscore `_` (Dart-style, COMPILER-ENFORCED, not mere convention) — NO `private` keyword. `_name` is private to the module and visible at every use site (`self._ptr`). Consequences:
- the ~266 now-redundant `pub` field annotations in vendor are removed;
- invariant types prefix their internals — e.g. Vec -> `_ptr: *T, _len: usize, _cap: usize` (corrupting `_len` is memory-unsafe, so it MUST stay hidden; with `unsafe` dropped, `_` privacy is now the ONLY thing protecting these invariants);
- plain-data / FFI structs construct by literal cross-module (`A { B: 0 }` now works with no constructor) — only `_`-private-field types need a `fn new()`;
- refines the mental anchor from "C++ CLASS" to "C++ STRUCT" (public fields + methods + const-correctness) — the `struct` keyword now tells the truth.
- (`_name` is an identifier with a leading underscore; bare `_` stays the wildcard/discard — different things.)

SYNTAX — not single keywords, but the same false-friend problem

# shadowing  (let x: i32 = 5; let x: bool = true;)
forbid same-scope re-declaration
Same-scope shadowing is pure Rust — C forbids redeclaring a name in one scope — and it lets a name silently change type; make a second `let`/`const`/`var` of the same name in the same block a hard error ("`x` already declared in this scope; pick a new name, or mutate the existing `let x`"). Nested-scope shadowing (`{ let x }` inside an outer `x`) STAYS — that is ordinary C block scoping, not a type swap. Verified: same-scope type-swap compiles today (bug), nested shadow correctly leaves the outer binding intact.

# <T>  generics
[T] (keep — already diverged)
C+ already uses square brackets `[T]` / `::[T]`; the residual trap is LLMs typing `<T>` — fix with a targeted error, not a syntax change.

# *const T / *mut T  raw pointer
*T (keep)
C+ has one raw-pointer form `*T`; LLMs from Rust write `*const`/`*mut` — diagnose, don't change.

# &[T]  slice
T[] (keep)
Postfix slice `T[]` is already non-Rust; no bare-reference slices to confuse.

# &str / String
str / Text  [[SUPERSEDED 2026-06-18: DROP the `as_str` method -> use a compiler COERCION (borrowed `Text` -> `str`, the `{ptr,len}` prefix) at arg/binding/return/receiver sites. PREREQUISITE: re-base the E0513 view-escape check off the `as_str` method NAME (sema.rs ~15030) onto the COERCION SITES, else coerced views escape unchecked. Region-lifetime syntax dropped, but the elision-grade view analysis is KEPT. See plan.md.]]
KEEP the split. `str` is a blessed borrowed view (`{ptr, len}`, Copy, no drop);
`Text` is the owned heap string (`{ptr, len, cap}`, non-Copy, Drop frees).
API decision: `Text::as_str(this) -> str` borrows without allocation;
`Text::clone(this) -> Text` copies/owns. Returning a local `Text` as `Text`
moves ownership and needs no clone. Do NOT add `to_str() -> str` as a copying
API unless `str` becomes owning; copied string data returns `Text`. With
`unsafe fn` gone, `as_str` becomes a normal view-producing method and
