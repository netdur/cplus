# metal_compute — GPU-via-FFI reference recipe

The v0.0.6 stdlib-model bet for GPU support: cpc embeds a precompiled
shader binary via `include_bytes!`, then dispatches the kernel through
ObjC interop into the Metal framework. No GPU backend lives in cpc;
the device code lives in `shaders/double.metal`, compiled by `xcrun
metal` to `shaders/double.metallib` at recipe build time.

The validation goal: prove that the v0.0.6 surface is *sufficient* —
that a real external `vendor/metal` package could be written today
against this exact pattern.

## What the kernel does

Doubles every element of a `[f32; N]` buffer on the GPU. A trivial
workload — the point is to exercise the full Metal command-queue +
command-buffer + compute-pipeline path, not to demonstrate a useful
algorithm. Real packages build on this skeleton.

## Prerequisites (macOS)

```bash
# One-time, ~1.5 GB:
xcodebuild -downloadComponent MetalToolchain
```

If `xcrun -sdk macosx metal --version` succeeds, you're set.

## Build + run

```bash
./build.sh
./metal_compute
```

`build.sh`:

1. Compiles `shaders/double.metal` → `shaders/double.air` → `shaders/double.metallib`
2. Lowers `src/main.cplus` to LLVM IR via `cpc --emit-ll`
3. Links with `clang -framework Metal -framework Foundation -lobjc`

Exits 0 when the GPU result matches the scalar reference for every
element of the input buffer.

## Why no `cpc build`

`cpc build` doesn't yet know about Apple framework search paths or
`-lobjc`. The `--emit-ll` + `clang` two-step is the documented escape
hatch for framework-linked binaries (cross-referenced from
[plan.md](../../../../plan.md) — see the §2.9 ObjC-interop note in
v0.0.4).


## What's in the C+ source

- A short list of ObjC selectors registered as `extern fn sel_*` calls.
- One `extern fn objc_msgSend` per typed return-shape (using
  `#[link_name = "objc_msgSend"]` to share the C symbol).
- `include_bytes!("../shaders/double.metallib")` to embed the kernel.
- The classic Metal dispatch sequence: device → queue → library (from
  the embedded bytes) → function → pipeline → command buffer →
  compute encoder → buffers → dispatch → commit → wait → readback.

The runtime touches the device side only via `*u8` raw pointers from
C+'s perspective — exactly the boundary the stdlib model commits to.
