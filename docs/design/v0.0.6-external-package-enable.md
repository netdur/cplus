# v0.0.6 — External-package enable: SIMD + GPU

**Status:** in flight (Slices 1A, 1B shipped 2026-05-20; Slice 1C decomposed, 1C.1 shipped; recipes + design doc in progress).
**Scope owner:** v0.0.6 release.
**Thesis in one line:** ship the *minimum* compiler surface that lets external packages do SIMD and GPU work, then stop.

---

## Why this design exists

The research note "Bringing AI and Compute Workloads to C+" (2026-05-19) estimated a major language-feature push to make C+ viable for ML and compute workloads — vector types, GPU kernel syntax, autograd, const generics for tensor shape, the works. The audit phase that followed cut ~80% of that estimate:

- **LLVM already has SIMD as IR primitives.** Fixed-width vector types (`<N x T>`), arithmetic that lowers to NEON / SSE / AVX, intrinsics (`llvm.fma`, `llvm.sqrt`, `llvm.minnum`, etc.), shuffles, masked loads — all of it. cpc doesn't need to *invent* SIMD; it needs to *expose* what LLVM already offers.
- **GPU work is FFI plus a shader binary.** Every GPU API — Metal, CUDA Driver, Vulkan — is a C ABI from the host side. The kernel itself is a precompiled binary blob (`.metallib`, `.cubin`, `.spv`) that the host loads at runtime. cpc doesn't need a device backend; it needs a way to *embed* the shader blob and a *raw-pointer FFI surface* to call into the host runtime.

The C+ stdlib precedent applies directly: the compiler ships primitives, the ecosystem ships libraries. v0.0.2 shipped the stdlib model. v0.0.5 shipped iterators-as-coroutines. v0.0.6 ships SIMD and GPU enablement the same way.

| LLVM capability | C+ surface | Status |
|---|---|---|
| Constant `[N x i8]` globals | `include_bytes!("path")` returning `*[u8; N]` | Shipped (Slice 1A) |

## What cpc plumbs (compiler-side work)

Listed in order of complexity, all delivered in v0.0.6:

1. **Parser + AST.** New `ExprKind::IncludeBytes { path }` (Slice 1A). Strict form: `include_bytes!(StringLit)`. The `!` marks this as the (currently sole) compiler-blessed macro shape — see §2.7's no-macros rule + open-question about future blessed builtins (`include_str!`, `env!`, ...).
2. **Sema (Slice 1A).** Path resolution relative to the source file containing the call, file read at type-check time, byte-length stored in `MonoInfo::include_bytes` (keyed by call span) so codegen materializes the constant global without re-reading. E0870 / E0871 / E0872 carry the resolved absolute path for actionable diagnostics.
3. **Sema (Slice 1B).** New `Ty::Simd { elem, lanes }` variant. `resolve_type` recognizes the new type names. `check_method_call` dispatches SIMD receivers through `check_simd_method_call`; `check_assoc_call` routes `f32x4::splat(...)` etc. through `check_simd_assoc_call`. New error codes: **E0873** (non-literal lane / shift count), **E0874** (lane / shift count out of range). The existing **E0410** (C-ABI export check) rejects SIMD types in `pub extern fn` signatures with a helpful "cast to `[T; N]` via `.to_array()`" hint.
4. **Codegen (Slice 1B).** `Ty::Simd` lowers to LLVM `<N x T>`. `Ty::Simd` joins the Copy set, the scalar-param set (`noundef`-eligible), the value-passed set, and a register-class layout (size = N × size(T), aligned to that pow-of-2). Method dispatch through `gen_simd_method_call` (instance) / `gen_simd_assoc_call` (associated). FMA / sqrt / abs / min / max / shuffle-broadcast / extractelement / insertelement / etc. all parametric over `(elem, lanes)` — adding a new width is two resolver entries plus per-width intrinsic decls.
5. **Codegen (Slice 1A).** Module-init pass walks the AST + sema's `IncludeBytesMap`, emits one `@.bytes.N = private unnamed_addr constant [LEN x i8] c"..."` global per unique resolved absolute path, populates a per-call-site lookup on `ModuleMetadata`. `gen_expr(ExprKind::IncludeBytes)` looks up the call's span and returns the global's address as the SSA value. Two calls with the same resolved path share one global.
6. **Codegen (Slice 1C.1).** Audited `is_scalar_ty` (the predicate gating `noundef` on value-passed params); widened to include `Ty::Simd` since vectors are register-class first-class scalars with no aggregate poison padding.

## What the ecosystem handles (in user-space packages, not cpc)

Everything not on the list above. Concretely:

| Capability | Lives in |
|---|---|
| SIMD algorithms (DSP filters, JSON SIMD, bytewise scanners, vectorized matmul, FFT) | External `vendor/simd_*` packages, written on top of cpc's `f32x4` / etc. surface. |
| GPU runtime wrappers (Metal, CUDA Driver, Vulkan) | External `vendor/metal`, `vendor/cuda`, `vendor/vulkan` packages — pure FFI to the host C ABI of each runtime. |
| Shader compilation | A separate build step. `.metal` → `.metallib` via `xcrun metal` + `xcrun metallib`. `.cu` → `.cubin` via `nvcc`. Compiled shaders embed via `include_bytes!`. cpc is not in this loop. |
| Tensor libraries (NumPy-shaped APIs, autograd, optimizer state, dataloaders) | External packages. Runtime shape via `Vec[usize]` strides — same as NumPy, PyTorch, every production framework. No const-generic shape tracking in cpc. |
| ML framework integration (PyTorch / JAX / XLA / TVM) | Out of scope. Those frameworks live in JIT design space; cpc is AOT only. The bridge would be FFI to their C APIs, exactly like any other C library. |

## What's NOT on the v0.0.6 roadmap (and why)

These were considered and explicitly cut. The rationale lives here so a future contributor can see why the gap exists rather than re-derive the analysis.

- **`#[align(N)]` attribute.** External package authors pad upload structs manually with explicit `_pad: u8` fields. Workable; not enabling. Revisit if SIMD or GPU package authors hit a real ergonomic wall (e.g. Metal's `MTLArgumentBuffer` 16-byte alignment requirement could surface this).
- **`addrspace(N)` source syntax.** Only relevant if cpc emits device-side code. The stdlib model has cpc emit host-side code only; GPU memory is opaque `*u8` from C+'s perspective.
- **Const generics for tensor shape.** External tensor packages use runtime shape via `Vec[usize]` strides, same as NumPy / PyTorch. AI-generated tensor code composes runtime shapes fine. Const generics would be a substantial language extension; we'll trade ergonomic shape inference for cpc-wide simplicity.
- **Inline asm.** Real but huge. Defer until a concrete workload (e.g. a specific NEON instruction LLVM won't auto-emit) demonstrates the need. Until then, `unsafe { extern fn ... }` to a hand-written `.s` file is the escape hatch.
- **GPU backend emission from cpc.** PTX, SPIR-V, AMDGPU — LLVM has the backends, but the surrounding pipeline (extract `#[kernel]` fns, compile separately, embed) is a v0.1.x architectural shift. Out of scope for v0.0.x. We embed precompiled binaries via `include_bytes!` instead — same effective outcome, far less compiler complexity.
- **JIT compilation.** cpc is AOT-only. ML frameworks that JIT (XLA, PyTorch 2, JAX, Mojo) live in a different design space. Cross-call into them via FFI when needed.
- **Autograd.** No source-to-source transformation in cpc (§2.7 forbids it). Future autograd story would be library-only, walking explicit `Function`-objects — but that's a stdlib question, not a compiler question.
- **Operator overloading exception for SIMD or tensor types.** Same as everywhere else: §2.6 holds. `a.add(b)` and `a.mul(b).fma(c, d)`, no exceptions. The audit-clarity argument from the research note applies directly to SIMD: method calls make ABI shape and allocation visible at the call site, whereas `a * b + c` hides the FMA peephole behind operator dispatch.
- **`include_str!` companion to `include_bytes!`.** Coupled to `Ty::Str`'s constructor story (fat-pointer aggregate); ship after `include_bytes!` if real demand surfaces. Workaround: `include_bytes!` then `str_from_raw_parts` inside `unsafe { ... }`.
- **`f32x16`, `f64x8`, AVX-512 / SVE2 widths.** Add when AVX-512 or SVE2 becomes a tier-1 target. Pure extension; doesn't break Phase 1.
- **SIMD shuffles, reductions, masked ops.** Foundation first; expansion in v0.0.7 once a real external SIMD package exists in tree to drive what shapes the user-side API actually wants (hand-rolled lane permutations vs. blessed `swizzle_xyzw` style helpers, single-mask vs. dual-mask blends, etc.).
- **Mask types** (`mask32x4`, `mask8x16`, etc.). Listed in the plan as Phase 1 widths but deferred — masked ops are tied to compare-and-blend patterns we haven't shipped yet. They land alongside the shuffle / reduction expansion in a follow-on slice.
- **TBAA, lifetime intrinsics, loop-unroll/vectorize attributes.** Originally bundled into Slice 1C; decomposed once attempted (see plan.md's 1C status table). 1C.1 (`noundef` widening) shipped. 1C.2 / 1C.3 / 1C.4 each need infrastructure refactors (per-scope alloca tracking, centralized `gen_load`/`gen_store` helper, statement-level attribute parsing) — they earn their own slices later in the v0.0.6 cycle.

## Open questions (do not block v0.0.6 ship)

1. **Blessed `vendor/simd` package?** The v0.0.2 stdlib precedent suggests one blessed external SIMD library is reasonable — it gives a stable place to land idioms like dot product, matrix multiply, masked select. Open whether it ships as `vendor/simd` alongside stdlib or remains purely external for the v0.0.7 cycle to discover the right API shape.
2. **Other macro-shaped builtins.** Is `include_bytes!`'s `!` syntax the only compiler-blessed macro we'll ever emit? §2.7 forbids user-defined macros but doesn't forbid compiler ones. Plausible candidates: `include_str!`, `env!("NAME")`, `concat!("a", "b")`, `cfg!(target_os = "macos")`. Decision deferred until we see a real use case for each.
3. **Whether to bless one mask type.** `mask32x4` (4-lane bool) is the obvious companion to `f32x4`. Deferring the full mask family doesn't prevent shipping a single `mask32x4` now if a shuffle / compare workload surfaces in 1B's expansion. Open through v0.0.6.
