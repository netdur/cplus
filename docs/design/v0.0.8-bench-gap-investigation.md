## Root-cause findings

Each finding is one specific IR-emission pattern, verified directly against the current codegen (`cpc --emit-ll`).

### Finding 1 — Duplicate field GEP+load per occurrence

Source: `v.x * v.x + v.y * v.y + v.z * v.z` (the raytracer Vec3 dot product).

Current cpc IR:

```llvm
%t2 = getelementptr inbounds %Vec3, ptr %v.addr1, i32 0, i32 0
%t3 = load float, ptr %t2, !tbaa !100001         ; v.x
%t4 = getelementptr inbounds %Vec3, ptr %v.addr1, i32 0, i32 0   ; v.x AGAIN
%t5 = load float, ptr %t4, !tbaa !100001         ; v.x again
%t6 = getelementptr inbounds %Vec3, ptr %v.addr1, i32 0, i32 1
%t7 = load float, ptr %t6, !tbaa !100001         ; v.y
%t8 = getelementptr inbounds %Vec3, ptr %v.addr1, i32 0, i32 1   ; v.y AGAIN
%t9 = load float, ptr %t8, !tbaa !100001
%t10 = fmul contract float %t7, %t9
%t11 = call contract float @llvm.fmuladd.f32(float %t3, float %t5, float %t10)
...
```

**What's wrong:** every `v.x` *expression* in source emits a fresh GEP+load pair, even when adjacent in the same expression. The redundant pair is folded by GVN downstream, but:

1. **IR bloats 3.3×** in `ray_color` (1,736 lines vs C's 518), slowing every LLVM pass.
2. **SLP-vectorizer pattern recognition is fragile** on interleaved-duplicate-load shapes. The May 17 cpc build emitted NEON 2-lane `fmul.2s` on the `(x, y)` pair; current build emits pure scalar. The bench.md note "lost auto-vectorization on Vec3 ops" maps directly to this.

**Fix:** cache field-read SSA values across one expression's codegen scope. `gen_field(receiver, name)` should hash `(receiver_place, name)` against an expression-local memo and return the cached SSA name when the same field is requested twice.

**Scope:** localized to `gen_field` + one small per-expression cache reset point. **Impact estimate:** closes most of the 4% raytracer gap; restores the May 17 NEON 2-lane behavior; shrinks `ray_color` asm proportionally.

**Site:** `cplus-core/src/codegen.rs:6936` (`fn gen_field`).

### Finding 2 — Struct-literal assignment to a place emits a redundant temp + full struct copy

Source: `table[idx] = Entry { key: k, val: v };` (the hashmap insert hot path).

Current cpc IR:

```llvm
%a10 = alloca %Entry                            ; temp for the literal
%t11 = getelementptr inbounds %Entry, ptr %a10, i32 0, i32 0
store i32 7, ptr %t11, !tbaa !100002           ; build field-by-field in temp
%t12 = getelementptr inbounds %Entry, ptr %a10, i32 0, i32 1
store i32 42, ptr %t12, !tbaa !100002
%t13 = load %Entry, ptr %a10                    ; load the whole struct
store %Entry %t13, ptr %t9                      ; copy to destination
```

**What's wrong:** the literal is built in a temp `%a10`, then `load %Entry, ptr %a10` → `store %Entry, ptr %t9` round-trips the aggregate through SSA. Two issues:

1. **The aggregate load + store carry no TBAA tag.** `tbaa_tag_for` returns `None` for `Ty::Struct(_)` (current code at `cplus-core/src/codegen.rs:140`). LLVM defaults to "may-alias-anything" on the copy, defeating alias analysis across the insert hot path.
2. **SROA usually folds the temp** at `-O2+`, but the pattern survives in edge cases (volatile-ish hot loops where SROA hits its budget; cross-BB stores).

**Fix:** when the destination of a `=` assignment is a place pointer (raw-pointer index, array index, field access), AND the RHS is a `StructLit` / `GenericStructLit`, emit field-by-field stores directly to the destination's field pointers. Skip the temp alloca entirely.

**Scope:** localized to `gen_assign` (or wherever assignment-to-place is handled). One new branch: if `op == Assign` and `value.kind == StructLit { fields }` and `target` resolves to a place pointer, GEP each destination field and `gen_store` directly.

**Impact estimate:** closes most of the 15% hashmap insert gap. Also tightens any "build a literal, assign to a slot" code path elsewhere (Vec3 construction, etc.).

**Site:** `cplus-core/src/codegen.rs::gen_assign` (and the StructLit branch of `gen_expr`).

### Finding 3 — `expr_value_ty_with_bindings` misses `Call`, `StructLit`, `Path`

Source: bench.md gotcha #4 — the mixed-if-arm panic.

```cplus
let result: V = if cond {
    v_make(3.0f32, 4.0f32)                // Call returning V
} else {
    let r_perp: V = ...;
    ...
    V { x: ..., y: ..., z: 0.0f32 }       // StructLit
};
```

cpc panics: `let init produces a value` at `codegen.rs:5902`.

**What's wrong:** `gen_if` decides whether to allocate a result slot by calling `block_value_ty_with_bindings(then) .or_else(|| expr_value_ty_with_bindings(else_branch))` (`codegen.rs:9466-9468`). Both helpers ultimately delegate to `expr_value_ty` (`codegen.rs:10988`), which has cases for `IntLit`, `FloatLit`, `BoolLit`, `Block`, `If`, `Binary`, `Unary`, `Path` (treats every enum as `i32`), `Cast` — and falls through to `_ => None` for **`Call`**, **`StructLit`**, `GenericStructLit`, `GenericEnumCall`, `TupleLit`, `Match`. So an if-expr where neither arm's tail expression matches the special-cased shapes returns no type, no result_slot gets allocated, and the let-init panics.

**Fix:** extend `expr_value_ty_with_bindings` (the `_with_bindings` variant — it has access to `self.types` and `self.fns`) to also handle:

- `ExprKind::Call { callee, type_args, .. }` — look up the callee's signature in `self.fns` / `self.fns_generic`, return the return type.
- `ExprKind::StructLit { name, .. }` — look up `self.types.struct_by_name.get(&name.name)`, return `Ty::Struct(id)`.
- `ExprKind::GenericStructLit { name, type_args, fields }` — look up the monomorphized struct in `self.types.struct_by_name` using the mangled name; sema's `MonoInfo::struct_instantiations` has the mapping.
- `ExprKind::Path { segments }` — sema handles `Color::Red`-shaped paths; for tagged-enum constructors with payload, return `Ty::Enum(id)` for the enum named in segment 0.

**Scope:** purely additive. Same helper, more arms.

**Impact estimate:** unblocks user code that builds struct results from mixed if-arms (raytracer's refract, every shader-style branch builder). Currently users work around by allocating a `let mut result: V = default; if … { result = …; }`.

**Site:** `cplus-core/src/codegen.rs:5494` (`fn expr_value_ty_with_bindings`).

### Finding 4 — TBAA does not tag aggregate (struct / array / SIMD) loads + stores

Source: every struct copy, every array element store through a typed pointer, every SIMD load/store.

Current cpc emits (from the struct-write example above):
```llvm
%t13 = load %Entry, ptr %a10                   ; NO !tbaa tag
store %Entry %t13, ptr %t9                     ; NO !tbaa tag
```

`tbaa_tag_for` (`codegen.rs:140`) explicitly returns `None` for `Ty::Struct(_)`, `Ty::Enum(_)`, `Ty::Array(_)`, `Ty::Simd { .. }`, `Ty::Str`, `Ty::String`, `Ty::Slice(_)`.

**What's wrong:** without `!tbaa` on the aggregate access, LLVM treats it as may-alias-anything. For hot insert loops where the only accesses are `*Entry` operations, the conservative analysis blocks LSE / DSE / load-PRE across the hot path.

**Fix:** generate a TBAA leaf per struct / enum / array type lazily, parented under the root. Same shape as the primitive leaves, keyed by the type's mangled name. SIMD types get one leaf per `(elem, lanes)` shape.

The "struct-path-aware" full TBAA (per-field nested type) is a v0.0.9+ exercise; the *whole-struct* leaf is enough to disambiguate `*Entry` from `*Bucket` from `*Sphere`.

**Scope:** the `match ty` in `tbaa_tag_for` (line 124–141). Replace the `_ => return None` with type-keyed lookups; add the per-type leaves to `tbaa_leaves`.

**Impact estimate:** partial mitigation of finding 2 even without the gen_assign rewrite — the aggregate copy still costs cycles, but at least LLVM can prove the source and destination don't alias other types' accesses.

**Site:** `cplus-core/src/codegen.rs:124` (`fn tbaa_tag_for`).

## Things bench.md flagged that are NOT actually regressions

### f32 literal ULP regression (bench.md gotcha #6)

bench.md claims `0.4f32` emits as f32 `0x3ECCCCCD` whereas C's `0.4f` is `0x3ECCCCCC`.

**Verified clean as of May 20 cpc**: the lexer fix in commit `8f46903` parses F32-suffixed literals directly to f32 before widening to f64-AST-storage; codegen's `(*v as f32 as f64).to_bits()` recovers the exact bits. The emitted IR `float 0x3FD99999A0000000` narrows to f32 `0x3ECCCCCD`, which is IEEE round-to-nearest-even of 0.4 — *identical* to what C's `0.4f` produces. bench.md's note appears to predate the fix.

(For posterity: 0x3ECCCCCC is one ULP below 0.4; 0x3ECCCCCD is the correct rounded value.)

### Lifetime intrinsics are not currently emitted in function bodies

v0.0.7 added the `declare void @llvm.lifetime.start.p0(i64, ptr)` preamble, but `cpc --emit-ll` shows zero `call void @llvm.lifetime.start` invocations in actual function bodies. The infrastructure is declared but unused — meaning v0.0.7 1.1 shipped the declaration but not the per-scope bracket emission, OR the per-scope emission is gated on something I haven't tracked down. **Worth checking, but it's not a regression: status quo = "neither help nor hurt," same as before v0.0.7.**

## Recommended fix order for v0.0.8 (or earlier)

1. **Finding 3 (mixed if-arm panic)** — Smallest patch, biggest user-visible win. Pure addition to one helper function. No risk of regression. **Do first.**
2. **Finding 4 (TBAA for aggregates)** — Localized to `tbaa_tag_for`. Adds metadata, never removes it. Marginal-to-meaningful perf win on alias-bottlenecked loops.
3. **Finding 1 (field-read caching)** — The raytracer NEON regression. Needs a per-expression memo of `(place, field)` → SSA name. Moderate scope; needs care to invalidate the cache across writes.
4. **Finding 2 (direct field-stores for struct literals)** — Hashmap insert gap. Larger change to `gen_assign`. Moderate-to-high impact.

After 1+2+3+4 the bench picture should look like:
- Raytracer: within 1% of C (closes the lost-NEON-2-lane gap).
- Hashmap insert: within 5% of Rust (skipping the aggregate temp + tagged aggregate copies).
- Hashmap lookup: stays ahead of C; may close some gap to Rust.
- JSON tokenizer: unchanged (already tied with C).

No language changes, no plan ordering risk — these are all codegen-only refactors in `codegen.rs`. Each is < 100 LOC except finding 1 which probably needs ~150 LOC including the cache invalidation logic.

## What this work does NOT include (deferred to v0.0.9+)

- **Struct-path-aware TBAA.** Per-field nested type leaves so `entry.key` and `entry.val` don't alias each other. Useful for SROA-resistant patterns but a v0.0.9 exercise.
- **Auto-vectorization hints on Vec3-shaped types.** Could expose `(x, y, z)` as a typed pair via `#[repr(packed)]` or `#[align(8)]`. Crosses into language design territory.
- **Hand-rolled SIMD raytracer port.** v0.0.8 plan Phase 1B already calls for this; would let us measure cpc SIMD-vs-scalar directly.
